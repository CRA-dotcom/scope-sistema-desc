Projex — Guía del Sistema

Cómo funciona el sistema hoy, paso a paso. Este documento es un README vivo: cada vez que cambie el flujo, se actualiza aquí.

📖 Ver guía de uso (UI paso a paso) → 📘 Estás en: guía técnica

Projex es un SaaS multi-tenant que automatiza la proyección financiera y la generación de entregables profesionales para firmas de consultoría multidisciplinaria. Un consultor crea proyecciones anuales por cliente, se generan asignaciones mensuales automáticamente, y la plataforma orquesta el ciclo cuestionario → cotización → contrato → entregable.

Vista rápida

El sistema modela el ciclo de un consultor con múltiples organizaciones (tenants vía Clerk), donde cada tenant atiende a sus propios clientes finales.

Stack real

  • Frontend: Next.js 15 (App Router) · React 19 · Tailwind 4 · (sin shadcn/ui instalado todavía)
  • Backend: Convex (DB + queries/mutations/actions + storage + cron)
  • Auth: Clerk Organizations (multi-tenant con orgId en JWT)
  • AI: Anthropic SDK → modelo claude-sonnet-4-20250514 desactualizado
  • Email: Resend
  • PDF: Puppeteer-core + @sparticuz/chromium vía /api/generate-pdf
  • Deploy: Vercel (primario) · Railway (secundario)

1. Auth + Organización

Todo acceso a la plataforma requiere estar autenticado con Clerk y pertenecer a una organización.

Usuario inicia sesión

Clerk maneja el /sign-in. Si el usuario tiene múltiples orgs, Clerk muestra selector.

src/middleware.ts:3-6 — rutas públicas: /sign-in(.*) y /q(.*). El resto pasa por auth.protect().

JWT llega a Convex con orgId

convex/auth.config.ts define el issuer de Clerk. El JWT trae orgId como claim (o org_id fallback).

Helpers de auth en Convex

convex/lib/authHelpers.ts expone:

  • requireAuth(ctx) — throw si no hay identidad.
  • getOrgId(ctx) — throw si no hay org.
  • getOrgIdSafe(ctx) — devuelve null para queries reactivas durante loading.
  • requireAdmin / requireSuperAdmin.

Toda query/mutation debe filtrar por orgId. Excepciones justificadas: queries públicas por accessToken (cuestionarios).

3. Proyección

La proyección es el corazón del sistema. Una proyección = un cliente + un año + configuración financiera + servicios activos. Al crearla se auto-generan 12 asignaciones mensuales por cada servicio activo.

Cómo se crea

Form en /proyecciones/nueva

El consultor ingresa: clientId, year, annualSales, totalBudget, commissionRate, seasonalityData (array de 12 meses), serviceConfigs (cada servicio: chosenPct + isActive).

Mutation projections.create

convex/functions/projections/mutations.ts:12-149

  1. Verifica que el cliente pertenece al org.
  2. Carga detalles de cada servicio (min/max/default %).
  3. Lee orgConfigs para modo de cálculo (weighted/fixed, commission mode, seasonality enabled).
  4. Llama a calculateProjection() del motor.
  5. Inserta row en projections con status draft.
  6. Por cada servicio → inserta projectionServices.
  7. Si el servicio está activo → inserta 12 filas en monthlyAssignments (una por mes) con status pending e invoice status not_invoiced.

Motor de cálculo

convex/lib/projectionEngine.ts — lógica pura, con tests.

  • Factor de Estacionalidad: FEmes = ventasMes / (ventasAnuales / 12)
  • Distribución: budgetServicio = remainingBudget × (chosenPct / sumPesosActivos)
  • Comisión proporcional: se calcula como % de ventas mensuales y se distribuye proporcionalmente.

Recalcular (ojo)

⚠️ Bug conocido: projections.recalculate (mutations.ts:267-295) borra TODAS las monthlyAssignments del servicio y las recrea con status: pending. Si el cliente ya tenía meses con invoiceStatus: paid o entregables delivered, esos estados se pierden. Fix pendiente: preservar filas con estado no-default.

4. Cuestionario Foco

Esta es la sección que pediste entender. Aquí está exactamente cómo y cuándo se generan los cuestionarios.

¿Cuándo se genera un cuestionario?

Nunca automáticamente. El cuestionario se genera cuando el consultor pulsa un botón desde la UI de una proyección. No hay cron, no hay trigger al crear cliente, no hay trigger al aprobar cotización.

Regla dura: solo puede existir UN cuestionario por proyección. Intentar generar otro tira error "Ya existe un cuestionario para esta proyección." (questionnaires/mutations.ts:42-44).

¿Cómo se genera?

Función: questionnaires.generate({ projectionId }) — convex/functions/questionnaires/mutations.ts:22-99

Validaciones

Verifica que la proyección pertenece al org del usuario. Verifica que no exista cuestionario previo. Verifica que haya al menos un projectionService activo (mutations.ts:54).

Construye las preguntas

Hay dos tipos:

A) 3 preguntas genéricas fijas (hardcoded, mutations.ts:7-20) — aplican a todos los servicios:

  • q_1_info_general — "Información general del servicio requerido"
  • q_2_documentos — "Documentos requeridos para este servicio"
  • q_3_observaciones — "Observaciones especiales o requerimientos adicionales"

B) 1 pregunta por servicio activo — generada en bucle:

q_svc_{projServiceId}_detalle
"Detalle específico para el servicio: {serviceName}"

Entonces: si la proyección tiene 5 servicios activos, el cuestionario tendrá 3 + 5 = 8 preguntas.

Genera el accessToken

mutations.ts:82-85 — concatena Math.random() + Date.now() + Math.random() (base36).

⚠️ Seguridad débil: este token no es criptográfico. Es adivinable con suficientes intentos y no tiene expiración. Reemplazar por HMAC firmado con secret + expiresAt es uno de los fixes pendientes del sprint v2.

Inserta el record

Crea la fila en questionnaireResponses con status: "draft", answer: "" en todas las preguntas, y el accessToken generado.

Cómo se comparte con el cliente

El consultor abre la vista del cuestionario y copia el link público:

https://tu-dominio.com/q/{accessToken}

Este link es público (definido en src/middleware.ts:5 como /q(.*)). No hay envío automático de email — el consultor debe compartir el link por el medio que quiera (WhatsApp, email manual, etc.).

Gap del sprint v2: enviar el cuestionario automáticamente al cliente por Resend con link firmado HMAC. Hoy no existe esa acción.

Cómo responde el cliente (flujo público)

Abre /q/[token]

src/app/q/[token]/page.tsx — carga el cuestionario vía publicQueries.getByToken (sin auth).

Ve las preguntas + branding del org

La query también carga el orgBranding (logo, colores) para presentar el cuestionario con identidad visual del consultor.

Responde y "Guardar progreso"

Llama publicMutations.updateResponsesByToken. Si el status era draft o sent, pasa automáticamente a in_progress.

"Enviar respuestas" (submit final)

Llama publicMutations.submitByToken. Status → completed + set completedAt.

Qué pasa cuando se completa

Al hacer submit desde la vista autenticada del consultor (questionnaires.submit, mutations.ts:160-202), el sistema programa un email de notificación al ejecutivo:

to: "ejecutivo@projex-platform.com"   // ← hardcoded placeholder
subject: "Cuestionario completado - {clientName}"
⚠️ Gap: el destinatario está hardcoded. Debería resolverse desde Clerk (assignedTo → email del ejecutivo asignado al cliente). Ver mutations.ts:193.

Nota: el submitByToken (que usa el cliente final) NO dispara el email. Solo lo hace cuando el consultor manualmente ejecuta submit. Ver inconsistencia en publicMutations.ts:37-55.

5. Cotización

Las cotizaciones son por servicio individual, no anuales agregadas (gap del sprint v2).

quotations.generate({ projServiceId })

convex/functions/quotations/mutations.ts:5-182

  1. Busca template activo: primero específico del servicio, luego genérico del org, luego global.
  2. Si hay template: resuelve variables {{name}}, {{rfc}}, {{annualAmount}}, etc.
  3. Si no hay template: usa HTML inline default.
  4. Inserta quotations con status: "draft".

Consultor edita (opcional)

updateContent solo funciona en draft.

Consultor genera PDF

UI llama POST /api/generate-pdf con el HTML + branding. Puppeteer renderiza. setPdfStorageId guarda el storage ID en la cotización.

Consultor marca sent

Manual. No hay envío automático de email con PDF adjunto. Gap del sprint v2.

Consultor marca approved o rejected

Botón manual. No hay landing público accept/decline con HMAC. Gap del sprint v2.

6. Contrato

Solo se puede generar un contrato desde una cotización con status: "approved" (contracts/mutations.ts:17-21).

contracts.generate({ quotationId })

Mismo patrón que cotización: busca template, resuelve variables, inserta con status: "draft".

PDF + envío manual

Mismo flujo que cotización.

Marcar "firmado"

Un click manual del consultor. No hay integración con MiFiel ni otra plataforma de firma electrónica. Gap crítico del sprint v2.

7. Entregable (pipeline AI)

Esta es la parte más automática del sistema. Un entregable se dispara manualmente desde una monthlyAssignment, pero una vez disparado, Claude hace el trabajo.

flowchart LR A["UI: Generar entregable"] --> B["action: generateDeliverable"] B --> C["Resolver variables
cliente / proyeccion / servicio"] C --> D["Buscar template"] D --> E["Claude llena variables AI
5 llamadas secuenciales"] E --> F["saveGenerated
status: pending"] F --> G["action: auditDeliverable"] G --> H["Claude audita"] H -->|approved| I["Usuario click Deliver"] H -->|rejected| J["retry o corregir manual"] I --> K["PDF + email cliente"] K --> L["assignment.status = delivered"]

Pipeline detallado

  • Modelo: claude-sonnet-4-20250514 (actualizar a 4-5)
  • Max retries: 3 con backoff exponencial (1s/2s/4s)
  • Cost tracking: cada llamada se guarda en deliverables.aiLog[] con tokens + USD
  • Pricing hardcoded: $3/1M input, $15/1M output
  • Prompt caching: NO implementado — pierdes 50-80% de ahorro
  • Llamadas en paralelo: NO — hoy son secuenciales
🔴 Blocker activo: ANTHROPIC_API_KEY no está configurada en Convex. Sin ella, getAnthropicClient() devuelve null y todas las variables AI se reemplazan por "[AI no disponible — configurar API key]". El audit se auto-aprueba. Los entregables hoy son cascarones. Fix: npx convex env set ANTHROPIC_API_KEY sk-ant-...

Estados del entregable

auditStatus:  pending → approved | rejected | corrected
deliveredAt:  se setea en deliver()
retryCount:   incrementa en cada regeneración

Al hacer deliver() (deliverables/mutations.ts:102-150):

  1. Verifica que audit está en approved.
  2. Set deliveredAt.
  3. Actualiza la monthlyAssignment a status: "delivered".
  4. Envía email al cliente (${rfc}@placeholder.com — gap).

Estados y transiciones (todas las tablas)

projections.status: draft → active → archived 
draft    : recién creada, editable
active   : en operación (cron puede actuar sobre ella)
archived : histórica, read-only
monthlyAssignments.status: pending → info_received → in_progress → delivered 
pending       : creada, aún sin info del cliente
info_received : cuestionario completado del cliente
in_progress   : entregable en generación
delivered     : entregable aprobado y entregado
monthlyAssignments.invoiceStatus: not_invoiced → invoiced → paid
questionnaireResponses.status: draft → sent → in_progress → completed
quotations.status: draft → sent → approved/rejected
contracts.status: draft → sent → signed/cancelled
deliverables.auditStatus: pending → approved/rejected/corrected
